# TZ19 Deep Audit (2026-03-04)

## Объём и метод
- Сверены 19 базовых ТЗ-документов в `docs/1_*.docx ... docs/19_*.docx`.
- Сверка делалась по трём осям:
  - функциональное соответствие (что реально есть в коде),
  - архитектурное соответствие (как это реализовано),
  - соответствие «духу» VITO (автономность, самообучение, экономичность, управляемость).
- Дополнительно учтены актуальные изменения сверх старого ТЗ (durable orchestration, tooling governance, self-learning pipeline, dashboard controls).

## Короткий вывод
- VITO **в целом соответствует духу ТЗ**, но не равномерно по всем блокам.
- Сильные стороны: оркестрация, memory/skills, управляемость через dashboard, safety/observability.
- Слабые стороны: живой browser execution на внешних сайтах, устойчивые live-check после ручного логина, «человечность» диалога в edge-кейсах, end-to-end боевые публикации на всех платформах.

## Оценка по 10-балльной шкале

### 1) Soul/Principles (docs 1, 6) — **8.3/10**
- Плюсы:
  - Финансовые лимиты, бюджетные гейты, cost/аномалии.
  - Контекст владельца, prefs, memory блоки, anti-pattern журнал.
- Минусы:
  - В некоторых сценариях коммуникация всё ещё «техническая» вместо дружелюбного результата.
  - Автономность в браузерных задачах ограничена реальными ограничениями платформ/сессий.

### 2) Core Architecture (docs 2, 5, 16) — **8.8/10**
- Плюсы:
  - `DecisionLoop` + durable `OrchestrationManager`, workflow sessions/interrupts.
  - Step contracts, retry-пайплайн, agent dispatch fallback.
- Минусы:
  - Часть «универсального browser executor» пока ближе к набору специализированных flow, чем к по-настоящему универсальному слою.

### 3) Roadmap/Execution Discipline (docs 3, 8, 19) — **8.1/10**
- Плюсы:
  - Есть боевой чеклист, фазы, evidence-driven тест-подход.
  - Идёт commit cadence и пакетная реализация.
- Минусы:
  - Не все боевые сценарии закрыты фактами на прод-подобных платформах.

### 4) Deploy/DevOps/Recovery (docs 4, 15, 17, 18) — **8.9/10**
- Плюсы:
  - Systemd/runtime, health/logs, recovery/matrix, rollback history.
- Минусы:
  - Browser runtime на сервере чувствителен к ресурсам ОС (пример: Chromium thread limits).

### 5) Judge/Safety/Security (docs 7, 12) — **8.7/10**
- Плюсы:
  - Policy gates, allowlist/budgets, guardrails, governance reports, key rotation workflows.
- Минусы:
  - Для user-facing UX часть блокировок/ограничений иногда выглядит «жёстко», не всегда прозрачно для владельца.

### 6) Knowledge/Monetization/Operations (docs 9, 10, 11, 13, 14) — **8.0/10**
- Плюсы:
  - База знаний платформ и режим Gemini-only уже есть.
  - Dry-run контуры публикации и social sdk присутствуют.
- Минусы:
  - Боевые publish/delete через browser на всех площадках не доказаны единым сквозным протоколом.

## Итоговые оценки
- **Соответствие духу идеи VITO:** **8.5/10**
- **Функциональная полнота относительно TZ19:** **8.2/10**
- **Текущая инженерная зрелость (надёжность/управляемость):** **8.7/10**
- **Итоговая интегральная оценка:** **8.4/10**

## Критичные гэпы до «боевого 10/10»
1. Универсальный browser live-execution слой с валидацией реальной авторизации (не только ручной флаг).
2. Единый протокол доказательств publish/edit/delete по каждой платформе (с артефактами и rollback).
3. Более устойчивый conversational NLU для коротких/кривых формулировок без ухода в «плановую канцелярщину».
4. Сквозной статус-модуль (цели/таски/финансы/вызовы агентов) без расхождений между карточками.
