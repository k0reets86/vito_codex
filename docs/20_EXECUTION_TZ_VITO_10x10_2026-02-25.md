# ТЗ: VITO 10/10 (исполняемый план)
Дата старта: 2026-02-25
Статус: ACTIVE
Владелец: VITO Core + агентная команда

## 1. Цель
Довести VITO до уровня 10/10 по блокам:
- Оркестрация агентов
- Память и навыки
- Диалог с владельцем
- Самообучение
- Публикации/коммерция
- Безопасность/контроль расходов
- Соответствие ТЗ

## 2. Принципы выполнения
- Никаких заявлений "сделано" без `ExecutionFacts` (evidence URL/ID/path/log).
- Любая интеграция внедряется через fallback (не ломаем прод).
- Сначала платформа/процесс/валидация, потом масштабирование.
- Минимум мусорных действий: одна цель/один поток/итеративная доработка.
- Дешевый роутинг моделей по умолчанию, эскалация только при необходимости качества.

## 3. Область работ (ближайший цикл)
Выполняем поочередно, с фиксацией фактов в чеклисте.

### Этап A. Платформенный execution-контур
1. Унифицировать `publish` pipeline по платформам с обязательным evidence contract.
2. Ввести предикаты готовности карточки продукта (title/description/media/tags/category/price).
3. Ввести `dry-run` и `verify-run` для публикаций.
4. Добавить блокировку редактирования существующих live-продуктов без явного запроса владельца.
5. Добавить контроль повторов/дубликатов публикаций.
6. Укрепить сценарии загрузки для Gumroad через browser-flow (с учетом лимитов платформы).

Критерий приемки:
- Пайплайн не репортит успех без верифицированного evidence.
- Любой провал фиксируется в failure memory с причиной и next action.

### Этап B. Управление и аналитика (Dashboard + DataLake + KPI)
1. Дашборд: операционные таблицы/действия (не JSON-виджеты).
2. Управление `.env` и переключателями с безопасной политикой секретов (write-only where needed).
3. KPI контур агентов: success rate, latency, retry rate, spend, ROI-метки задач.
4. DataLake: нормализация событий (action/decision/error/cost/approval/evidence).
5. Динамика KPI (день/неделя/30 дней) + сигнал деградации качества.

Критерий приемки:
- Из UI можно включать/выключать ключевые режимы и видеть состояние исполнения.
- KPI даёт практический рейтинг агентов и приоритеты улучшения.

### Этап C. Надежность и безопасность
1. Глобальный anti-hallucination gate для всех исходящих каналов.
2. Глобальная защита от prompt injection для внешних данных.
3. Self-heal контур: fix -> test -> rollback (enforced policy).
4. Сетевой watchdog + graceful fallback коммуникации владельца.
5. Контроль расходов: policy-guard + лимиты + отчетность по роутингу моделей.

Критерий приемки:
- Нет неподтвержденных "успехов" в коммуникации.
- При регрессии код автоматически откатывается и фиксируется причина.

### Этап D. Память/навыки/самообучение
1. Двухконтурная память: `what_works` и `what_fails`.
2. Автообновление playbooks агентов из практики (после верифицированных кейсов).
3. Skill registry: версия, совместимость, риски, покрытие тестами.
4. Обновление базы знаний платформ/моделей по расписанию.

Критерий приемки:
- В следующих похожих задачах VITO выбирает ранее успешный путь без лишних поисков.

## 4. Что не делаем в этом цикле
- Не разворачиваем "широкий каталог 3000+" без стандарта качества/валидации.
- Не добавляем новые платформы без capability matrix и acceptance tests.

## 5. Definition of Done (DoD)
- Тесты: `pytest -q -c /dev/null tests -k "not integration_offline"` проходят.
- Сервисы поднимаются без crash loops.
- Каждая завершенная задача имеет evidence.
- В чеклисте отмечены статус, дата, ссылка на изменение/лог/тест.

## 6. Ближайший порядок выполнения
1) Зафиксировать ТЗ/чеклист и baseline.
2) Закрыть Этап A.
3) Закрыть Этап B.
4) Закрыть Этап C.
5) Закрыть Этап D.
6) Финальный глубокий аудит + сравнение с OpenClaw по capability matrix.
